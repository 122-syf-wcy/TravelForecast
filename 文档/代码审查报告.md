
## 四、🟢 潜在Bug

### 4.1 Gateway CORS配置冲突

- **文件**：`TravelForecastGateway/application.yml` 第46行
- **问题**：同时配置了 `"*"` 和具体域名，且 `allow-credentials: true`
- **影响**：当 `allow-credentials=true` 时，`allowed-origins` 不能为 `"*"`，浏览器会拒绝
- **建议**：移除 `"*"`，只保留具体域名；或生产环境关闭 `allow-credentials`

### 4.2 Gateway StripPrefix配置问题

- **文件**：`TravelForecastGateway/application.yml` 第67行
- **问题**：`business-service` 路由的 `StripPrefix=1` 会去掉 `/api` 前缀，但后端 Backend 的 `context-path` 正好是 `/api`
- **影响**：请求 `/api/auth/login` 经Gateway后变成 `/auth/login`，但后端期望路径是 `/api/auth/login`
- **建议**：要么去掉 StripPrefix，要么后端去掉 context-path

### 4.3 小程序后端 `sql.init.mode=always`

- **文件**：`TravelForecastMiniProgramBackend/application.yml` 第13-15行
- **内容**：`mode: always` + `schema-locations: classpath:schema.sql`
- **影响**：每次启动都会执行建表SQL，如果SQL不是幂等的，可能导致数据丢失或报错
- **建议**：改为 `mode: never` 或确保SQL使用 `CREATE TABLE IF NOT EXISTS`

### 4.4 小程序后端与AI后端Redis库冲突

- **小程序后端**：`database: 1`
- **AI后端**：`database: 1`
- **影响**：两个服务共用同一个Redis数据库，可能存在Key冲突
- **建议**：小程序后端改为 `database: 3`

### 4.5 前端 `.env` 文件缺失

- **位置**：`TravelForecastFrontend/web/` 目录下没有 `.env` 文件
- **影响**：`VITE_API_BASE_URL` 未配置，默认使用 `/api` 相对路径
- **建议**：创建 `.env.development` 和 `.env.production`，明确API地址

### 4.6 数字人 `main.js` 中TODO未完成

- **文件**：`TravelForecast-DigitalHuman/src/main.js` 第185行
- **内容**：`// TODO: 将音频数据传给LipSyncEngine`
- **影响**：音频数据未正确传递给口型同步引擎
