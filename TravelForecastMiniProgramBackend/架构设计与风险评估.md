# 《智教黔行·智游六盘水》微服务架构设计与风险评估

> 评估日期：2026-02-07  
> 项目：TravelForecast 旅行预测系统 — 小程序后端拆分方案

---

## 一、现有 Web 后端现状分析

### 1.1 项目规模

| 指标 | 数量 | 说明 |
|------|------|------|
| 控制器 (Controller) | **54 个** | 涵盖管理后台、Web前台、商家端 |
| 实体类 (Entity) | **69 个** | 数据模型庞大 |
| 服务层 (Service) | **71 个** | 业务逻辑复杂 |
| Mapper | **50 个** | 数据访问层 |
| API 端点 | **200+** | 已属于大型单体应用 |

### 1.2 技术栈

- **框架**：Spring Boot 3.2.0 + Java 17
- **ORM**：MyBatis-Plus 3.5.5
- **数据库**：MySQL (HikariCP 连接池)
- **缓存**：Redis
- **AI**：通义千问 (DashScope SDK)
- **存储**：阿里云 OSS
- **地图**：高德地图 API
- **预测**：Python 预测服务 (localhost:8001)
- **认证**：JWT + 角色权限体系 (USER / MERCHANT / ADMIN)

### 1.3 已有微信小程序代码

Web 后端中已包含以下小程序相关模块，需**迁移到新后端后删除**：

| 模块 | 文件 | 说明 |
|------|------|------|
| 微信登录控制器 | `WechatLoginController` | `/api/auth/wechat/login` 等3个接口 |
| 微信服务 | `WechatService` / `WechatServiceImpl` | 登录、解密、签名验证 |
| 微信用户实体 | `WechatUser` | openid、unionid、session_key |
| 微信配置 | `application.yml` → `wechat.miniapp` | appid、appsecret |

## 二、性能风险评估

### 2.1 关键配置瓶颈

| 配置项 | 开发环境 | 生产环境 | 风险等级 | 说明 |
|--------|---------|---------|---------|------|
| 数据库最大连接数 | 20 | **10** | 🔴 极高 | Web后台+Web前台+小程序同时请求将耗尽连接 |
| 数据库最小空闲连接 | 5 | 5 | 🟡 中等 | 突发流量时连接创建延迟 |
| Redis 最大连接数 | 50 | **8** | 🔴 极高 | 缓存失效时大量请求打穿到数据库 |
| Redis 最大空闲连接 | 20 | **8** | 🟡 中等 | 连接复用率低 |
| 异步线程池核心线程 | 10 | 10 | 🟡 中等 | AI调用+文件上传等异步任务将争抢线程 |
| 异步线程池最大线程 | 50 | 50 | 🟢 尚可 | 但AI单次60s超时会长期占用 |
| HTTP连接池最大连接 | 200 | 200 | 🟢 尚可 | 外部API调用能力足够 |
| 文件上传限制 | 50MB | **10MB** | 🟢 合理 | 生产环境限制适当 |

### 2.2 并发风险场景分析

#### 场景一：AI 服务阻塞 (🔴 高危)

```
Web端用户A  ──→ AI聊天规划 (耗时30s) ──→ 占用线程1 + 数据库连接1
Web端用户B  ──→ AI聊天规划 (耗时45s) ──→ 占用线程2 + 数据库连接2
小程序用户C ──→ AI数字人讲解 (耗时20s) ──→ 占用线程3 + 数据库连接3
小程序用户D ──→ AI研学答题 (耗时15s)  ──→ 占用线程4 + 数据库连接4
...
═══════════════════════════════════════════════════════════
结果：核心线程池(10)被AI请求占满 → 普通API排队 → 页面卡顿/超时
```

#### 场景二：数据库连接耗尽 (🔴 高危)

```
生产环境数据库连接数：10
├── Web管理后台（管理员操作）    → 常占 2-3 个连接
├── Web前台（浏览景区/评价）     → 常占 2-3 个连接
├── Python预测服务回调           → 占用 1-2 个连接
├── 定时任务/异步任务            → 占用 1-2 个连接
└── 剩余给小程序的连接           → 仅剩 0-3 个！

小程序预期用户量远高于Web端（C端用户）→ 连接不够用 → 请求超时
```

#### 场景三：单点故障传播 (🔴 高危)

```
小程序文创电商搞促销活动
  → 瞬时并发激增（下单、查库存、查物流）
  → 数据库连接池打满
  → Web管理后台无法访问（管理员无法操作）
  → Web前台景区信息无法加载（用户体验崩塌）
  → 全线崩溃 ❌
```

### 2.3 风险评估结论

| 评估维度 | 评级 | 说明 |
|---------|------|------|
| 数据库连接承载 | 🔴 不可接受 | 生产环境10个连接无法同时支撑3个客户端 |
| AI服务并发 | 🔴 不可接受 | 长耗时AI请求会阻塞全局线程池 |
| 故障隔离性 | 🔴 不可接受 | 任一平台异常将影响其他所有平台 |
| 部署灵活性 | 🟡 较差 | 小程序任何改动需重新部署整个后端 |
| 代码维护性 | 🟡 较差 | 54个控制器已臃肿，再增加将难以维护 |
| 扩展弹性 | 🔴 不可接受 | 单体应用无法针对小程序端单独扩容 |

**最终结论：❌ 不建议在现有 Web 后端上直接叠加小程序业务，必须拆分独立后端。**

## 四、拆分后的性能对比

### 4.1 资源分配优化

| 配置项 | 拆分前(共用) | Web后端(拆分后) | 小程序后端(拆分后) |
|--------|-------------|----------------|------------------|
| 数据库连接数 | 10(共用) | 15(独占) | 15(独占) |
| Redis 连接数 | 8(共用) | 10(独占) | 10(独占) |
| 线程池核心 | 10(共用) | 10(独占) | 15(独占) |
| 线程池最大 | 50(共用) | 30(独占) | 50(独占) |
| AI请求隔离 | ❌ 互相阻塞 | ✅ Web端AI独立 | ✅ 小程序AI独立 |
| 内存占用 | ~1.5GB | ~1GB | ~800MB |

### 4.2 故障隔离效果

```
拆分前（单体）:
  小程序促销流量激增 → 数据库连接耗尽 → Web后台也崩溃 ❌

拆分后（微服务）:
  小程序促销流量激增 → 小程序后端连接池满 → 仅小程序降级
                                          → Web后台不受影响 ✅
                                          → Web前台不受影响 ✅
```
